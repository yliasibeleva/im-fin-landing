# CLAUDE.md — Шпаргалка по проекту

## Кто клиент

**Компания:** Империя Финанс  
**Сфера:** Бухгалтерский аутсорсинг (бухучёт, кадры, налоги, воинский учёт)  
**Мессенджер:** Max (основной, 90%) + Telegram (10%)  
**У каждого клиента:** группа в Max с бухгалтером + 1С:Предприятие

---

## Зачем этот проект (бизнес-контекст)

До бота всё велось вручную — бухгалтеры сами следили за сроками, доп. работы нигде не фиксировались, руководитель не видел кто что делает и не мог объективно оценить качество работы.

| Проблема | Решение в боте |
|----------|---------------|
| Пропуск сроков сдачи отчётности | Автоматические напоминания за 14/7/3/1 день в группу клиента |
| Нет контроля над бухгалтерами | Задачи с дедлайнами, статусами, KPI |
| Доп. работы не учитываются и не оплачиваются | Журнал доп. работ → Excel в конце месяца |
| Непонятно что входит в абонплату | Стандарт работы закреплён за каждой компанией |
| Нет оценки продуктивности | KPI: % выполнения, просрочки, ошибки, часы |
| Бухгалтеры не видят общую картину | Утренняя сводка дедлайнов в общую группу бухгалтеров |

## Этап 1 — Max-бот (buh-bot/)

### Стек
- Python 3.11+, maxapi (polling), SQLite, APScheduler, openpyxl

### Что умеет
- **Компании:** добавление/редактирование, ИНН, СНО (ОСН/УСН/ЕСХН/ПСН), тип (ООО/ИП), сотрудники, воинский учёт, стандарт работы, привязка группы Max
- **Дедлайны:** автогенерация по налоговому календарю РФ при добавлении компании, произвольные дедлайны, отметка выполнения
- **Задачи:** постановка бухгалтеру, приоритет, срок, статус
- **Доп. работы:** фиксация (тип, часы, сумма) сверх абонплаты
- **Ошибки:** журнал ошибок бухгалтеров → влияет на KPI
- **Напоминания:** ежедневно в 9:00 МСК — за 14/7/3/1 день в группу клиента и бухгалтеру лично
- **Отчёты:** текстовые + Excel (доп. работы + KPI бухгалтеров)

### Роли
| Роль | Как определяется |
|------|-----------------|
| Руководитель | max_user_id в ADMIN_IDS (.env) |
| Бухгалтер | Запись в таблице accountants |
| Клиент | Участник группы в Max |

### Ключевые команды

Руководитель (личка):
```
/start          — главное меню
/add_company    — добавить компанию (9-шаговый диалог)
/edit_company   — редактировать компанию
/company_info   — карточка компании
/add_accountant — добавить бухгалтера
/add_task       — поставить задачу
/add_deadline   — произвольный дедлайн
/add_error      — ошибка бухгалтера
/add_work       — доп. работа
/upcoming       — дедлайны на 30 дней
/myid           — узнать свой ID
```

Бухгалтер (личка):
```
/start          — меню бухгалтера
/mytasks        — мои задачи
/mydeadlines    — мои дедлайны
/add_work       — доп. работа
/myid           — узнать свой ID
```

В группе клиента:
```
/myid           — ID группы (для привязки к компании)
/status         — ближайшие дедлайны компании
```

### Таблицы БД
- `companies` — клиенты
- `accountants` — бухгалтеры
- `report_deadlines` — дедлайны (автогенерация + ручные)
- `tasks` — задачи
- `additional_works` — доп. работы
- `accountant_errors` — журнал ошибок
- `reminder_log` — лог отправленных напоминаний (защита от дублей)

### Запуск
```bash
cd buh-bot
pip install -r requirements.txt
python main.py
```
Токен бота: см. `.env` (не хранить в коде и документации!)  
После запуска: написать боту `/myid` → вставить ID в `.env` как `ADMIN_IDS`

### Статус деплоя
- **Сейчас:** бот на модерации в Max, ещё не активен
- **Хостинг:** не настроен (бот работает только пока открыто окно)
- **Планируется:** перенос на VPS / systemd-сервис для постоянной работы
- **Автозапуск:** не настроен
- **После модерации:** заполнить ADMIN_IDS в `.env` через `/myid`, запустить `python main.py`

### Карта handlers.py (61 KB, главный файл)

Два центральных хендлера — всё роутится внутри них:

**`text_handler`** (строка ~297) — обрабатывает все текстовые сообщения по состоянию пользователя:
| Состояние | Что собирает |
|-----------|-------------|
| `ADD_CO_NAME / INN / GROUP / STANDARD` | Диалог добавления компании (шаги 1–4) |
| `ADD_ACC_NAME / MAX_ID` | Добавление бухгалтера |
| `ADD_WORK_DESC / HOURS / AMOUNT / DATE` | Фиксация доп. работы |
| `ADD_TASK_TITLE / DESC / DUE` | Постановка задачи |
| `ADD_DL_NAME / DATE / PERIOD` | Произвольный дедлайн |
| `EDIT_CO_VALUE` | Редактирование поля компании |
| `ADD_ERR_DESC / DATE` | Запись ошибки бухгалтера |

**`callback_handler`** (строка ~489) — обрабатывает все нажатия кнопок по префиксу payload:
| Префикс payload | Действие |
|----------------|---------|
| `tax:` | Выбор системы налогообложения |
| `org:` | Выбор типа организации (ООО/ИП) |
| `emp:` / `mil:` | Да/нет по сотрудникам и воинскому учёту |
| `acc:` | Выбор бухгалтера при добавлении компании (шаг 5/9) |
| `task_acc:` | Выбор бухгалтера при постановке задачи |
| `gen_dl:` | Подтверждение генерации дедлайнов |
| `done_dl:` | Отметить дедлайн выполненным |
| `done_task:` | Отметить задачу выполненной |
| `work_co:` | Выбор компании для доп. работы |
| `work_type:` | Тип доп. работы |
| `task_co:` | Выбор компании для задачи |
| `task_acc:` | Выбор бухгалтера для задачи |
| `prio:` | Приоритет задачи |
| `dl_co:` | Выбор компании для дедлайна |
| `dl_type:` | Тип дедлайна |
| `edit_co:` | Выбор компании для редактирования |
| `edit_field:` | Выбор поля для редактирования |
| `edit_acc:` | Назначение бухгалтера при редактировании |
| `deactivate:` | Деактивация компании |
| `err_acc:` | Выбор бухгалтера для ошибки |
| `err_co:` | Выбор компании для ошибки |
| `co_info:` | Карточка компании |
| `menu:` | Навигация по разделам меню |
| `report:` | Генерация Excel-отчёта |

## Правила кода и ограничения

Детальные инструкции — в [buh-bot/CLAUDE.md](buh-bot/CLAUDE.md).  
Главное:
- Один `@dp.message_created` для текстов, один `@dp.message_callback` — вся маршрутизация внутри
- Только polling, только синхронный sqlite3 через `asyncio.to_thread`
- Состояния — только в памяти (StateStorage), не в БД
